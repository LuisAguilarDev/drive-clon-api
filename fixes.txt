I've now reviewed the entire FastAPI codebase. Overall
  this is a well-architected project — the layering is clean
  and consistently applied, multi-tenant isolation is
  disciplined, and the documentation is excellent. But there
  are several real anti-patterns worth fixing. Here's my
  assessment, ordered by severity.

  ---
  🔴 High severity

  1(Fixed). Broken transaction boundaries — every repository
  commits independently

  This is the most important issue. Every repository method
  calls self.db.commit() itself. That means any service
  operation spanning more than one repo call is not atomic:

  - FilesService.delete_folder (files_service.py:201-202):
  await self._files.soft_delete_in_folders(folder_ids,
  user.org_id)   # commit #1
  await self._folders.soft_delete_in_ids(folder_ids,
  user.org_id)     # commit #2
  - If commit #2 fails, files are trashed but their folders
  stay active → corrupted tree.
  - restore_folder (:242-245): three separate commits
  (restore_in_ids, restore_in_folders, reattach).
  - _purge_folder (:300-304): MinIO delete, then two
  separate commits.
  - EnsureOrganizationService.ensure: Keycloak org create →
  Postgres org create (commit) → set_org (commit) → root
  folder create (commit). A crash mid-way leaves an orphaned
  Keycloak org or a user without a root folder.

  Fix: the unit-of-work belongs at the request/service
  boundary, not inside repositories. Repositories should
  await self.db.flush() (to get generated IDs) and the
  service (or the get_db dependency) should own the single
  commit(). Concretely: remove commit() from repos, and
  commit once at the end of the dependency:
  async def get_db():
      async with SessionLocal() as db:
          try:
              yield db
              await db.commit()
          except Exception:
              await db.rollback()
              raise
  This is the single change with the highest payoff.

  2(Fixed). Leftover GCP/Firebase service-account key + unused
  Google Cloud dependency stack

  app/core/serviceAccountKey.json is a real service-account
  key file (dated Mar 2025) that nothing in the code
  references (grep found zero usages). Correspondingly,
  requirements.txt pulls an entire unused Google stack:
  google-cloud-firestore, google-cloud-storage,
  google-api-python-client, google-auth, firestore, grpcio,
  etc. — none imported anywhere.

  It's gitignored today, but a stray secret on disk is a
  liability (one .gitignore edit away from being committed),
  and the dead deps bloat the image and attack surface.
  Delete the file and prune the Google packages (also
  GitPython, passlib, bcrypt, psycopg2-binary, and PyJWT
  look unused — you use python-jose and asyncpg). Verify
  with a grep before removing each.

  3. Everything is buffered fully in memory — no streaming,
  no size limits

  For a "Drive clone", file handling never streams:
  - Upload: data = await file.read() (files.py:224) loads
  the whole file into RAM, then hands the bytes to MinIO.
  - Download: get_object reads the entire object into memory
  and returns a plain Response.
  - Folder ZIP: built entirely in an in-memory io.BytesIO
  (files_service.py:171-179), then getvalue() copies it
  again.

  There's also no max-upload-size validation at the boundary
  — a single large upload can OOM the worker. Acceptable
  for a POC, but flag it: prefer StreamingResponse + MinIO's
  streaming get for downloads, stream uploads to MinIO, and
  enforce a size cap in upload_file.

  ---
  🟡 Medium severity

  4. Repeated try/except boilerplate — DRY violation

  The exact try: ... except ResourceNotFound: raise
  HTTPException(404) from exc block is duplicated in ~12
  endpoints in files.py. Centralize it with FastAPI
  exception handlers:
  @app.exception_handler(ResourceNotFound)
  async def _not_found(request, exc):
      return JSONResponse(status_code=404,
  content={"detail": str(exc)})

  @app.exception_handler(OperationNotAllowed)
  async def _bad_request(request, exc): ...
  Then every endpoint loses its try/except and just calls
  the service. Big readability win.

  5. Manual service construction instead of FastAPI DI

  _build_service(db) (and the inline
  EnsureOrganizationService(...)/AccountService(...) in
  auth.py) wire dependencies by hand inside the route.
  Idiomatic FastAPI uses Depends providers:
  def get_files_service(db: db_dependency) -> FilesService:
  ...
  ServiceDep = Annotated[FilesService,
  Depends(get_files_service)]
  Beyond cleanliness, this is what lets tests swap
  implementations via app.dependency_overrides — currently
  impossible.

  6. Unsynchronized global JWKS cache

  _jwks_cache in security.py is a module global mutated
  without a lock. Concurrent first requests can fire
  concurrent JWKS fetches, and the cache has no time-based
  expiry (only refreshes on an unknown kid). Low blast
  radius, but wrap the fetch in an asyncio.Lock and consider
  a TTL.

  7. Recursive per-folder queries (N+1 on the tree)

  _collect_subtree_ids, _collect_files, and
  _collect_subtree_ids_any_state issue one query per folder
  as they recurse. Deep/wide trees → many round-trips per
  request (download ZIP, delete, restore, purge). A single
  recursive CTE would collapse each to one query. Fine at
  POC scale; note it before it grows.

  ---
  🟢 Low / nits

  - Model file & class naming breaks PEP 8. Files.py,
  Users.py, etc. should be lowercase modules
  (file.py/user.py), and entity classes are conventionally
  singular (User, not Users). Watch the collision with the
  files.py route module if you rename.
  - No tests, no linter (acknowledged in AGENTS.md). Given
  finding #1, this is exactly the category of bug a single
  integration test would catch. Worth adding ruff + a few
  pytest + httpx.AsyncClient tests around the trash/restore
  flows.
  - CORS uses allow_methods=["*"], allow_headers=["*"] with
  allow_credentials=True. Origins are properly restricted
  (good), so this is minor — but consider narrowing
  methods/headers.
  - config.py double-loads env: it calls load_dotenv() and
  sets env_file in SettingsConfigDict — redundant;
  pydantic-settings handles it alone.

  ---
  ✅ What's genuinely good

  Worth saying, since the structure is solid: strict
  one-directional layering that's actually respected;
  gateways as ABCs with the code depending on the interface;
  tenant isolation applied uniformly (org_id on every
  query, object keys prefixed by org, cross-tenant → 404
  without leaking existence); the single-status-enum
  lifecycle model is clean; migrations kept out of the app
  process; idempotent provisioning with the sub-relink
  fallback; consistent constructor injection. The bones are
  right — the issues above are mostly about transaction
  integrity and trimming dead weight, not redesign.

  ---
  Want me to fix any of these? The highest-value,
  lowest-risk batch I'd suggest: #1 (transaction
  boundaries), #2 (delete the key + prune deps), and #4
  (centralized exception handlers) — together they remove
  the real correctness bug and a chunk of boilerplate. I can
  do them as separate, reviewable changes.
